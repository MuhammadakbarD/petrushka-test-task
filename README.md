#  Задание 1 — Анализ требований

##  Выявленные проблемы

### 1. Противоречие количества

* П.2: минимум 1
* П.9: допускается 0

 Проблема: конфликт бизнес-логики

---

### 2. Противоречие цены

* П.7: цена фиксируется
* П.13: цена обновляется

 Проблема: невозможно одновременно фиксировать и обновлять цену

---

### 3. Неконкретная ошибка

> "Лимит корзины превышен"

 Проблема: не ясно какой лимит нарушен

---

### 4. Неполные требования по рекламе

 Нет:

* точного времени
* количества
* условий показа

---

##  Исправленные требования

### Ограничения корзины

| Ограничение   | Значение |
| ------------- | -------- |
| Один товар    | 1–10     |
| Разные товары | ≤ 5      |
| Всего товаров | ≤ 20     |

---

### Поведение системы

1. Пользователь может добавить товар в корзину
2. Количество можно изменять в диапазоне **1–10**
3. Удаление — через кнопку
4. При превышении лимитов отображаются конкретные ошибки
5. Цена фиксируется при добавлении

---

### Отображение

* название
* количество
* цена
* сумма
* итог

---

### Реклама

* будние дни
* 08:00–11:00 / 18:00–21:00
* максимум 3 блока

---

##  User Story

Как пользователь
я хочу добавлять товары в корзину
чтобы оформить заказ

---

##  Acceptance Criteria

* товар добавляется в корзину
* количество не превышает лимиты
* отображается корректная цена
* корзина корректно считает итог

---

##  Вопросы заказчику

* хранится ли корзина без авторизации
* что делать при отсутствии товара
* нужна ли персонализация рекламы
* можно ли объединять одинаковые товары

---

##  Нефункциональные требования

* Время ответа API ≤ 300 мс
* Доступность ≥ 99.9%
* Поддержка ≥ 10 000 пользователей
* Безопасность: HTTPS, авторизация

#  Задание 3 — Архитектура PUSH

## Архитектура

```
Mobile App
    ↓
API Gateway
    ↓
Notification Service
    ↓
Message Broker (Kafka / RabbitMQ)
    ↓
Push Service
    ↓
FCM / APNS
    ↓
User Device

Источники событий:
- Order Service
- Cart Service
- Marketing Service
```

---

## Поток данных

1. Событие возникает:

   * брошенная корзина
   * отмена заказа
   * маркетинг

2. Сервис отправляет событие в брокер

3. Notification Service обрабатывает

4. Push Service отправляет PUSH

---

##  Обоснование архитектуры

* асинхронность
* масштабируемость
* слабая связность сервисов
